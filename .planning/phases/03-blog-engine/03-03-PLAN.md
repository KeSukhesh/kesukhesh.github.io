# Plan 03-03: Search (CoVe Required)

## Goal
Add client-side fuzzy search to the blog using Fuse.js. Build search index at build time, create a search component as an Astro island.

## CoVe Trigger
Client-side Fuse.js with Astro island hydration — requires verification of:
1. Search index generated correctly at build time
2. Fuse.js hydrates via `client:load`
3. Search results render correctly
4. No hydration mismatch or SSR issues

## Tasks

### Task 1: Search Component
<task>
Create `src/components/BlogSearch.astro`:
- Accept props: posts array (title, description, tags, slug)
- Render a search input and results container
- Use a `<script>` tag (not client:load island) for simplicity — vanilla JS with Fuse.js
- On input, search with Fuse.js and render matching results
- Show results as clickable post links
- Hide results when input is empty
- Style to match design system
- Import Fuse.js in the script

Alternative approach: Since this is a static site, embed the search index as JSON in a script tag and use Fuse.js from CDN or bundled import. Astro's script bundling handles this well.
</task>
<verify>
- Component renders search input
- Typing filters posts correctly
- Clicking a result navigates to the post
- Empty input hides results
</verify>

### Task 2: Integrate Search into Blog Index
<task>
Update `src/pages/blog/[...page].astro`:
- Import and render BlogSearch component at top of page
- Pass all posts data (title, description, tags, id) as JSON for the search index
</task>
<verify>
- Search appears on blog index page
- Build succeeds
</verify>

## Verification
<verification>
1. `npm run build` completes without errors
2. Search input renders on blog index
3. Fuse.js is bundled correctly (check dist output)
4. Search functionality works client-side
</verification>

## CoVe Verification Steps
1. **Baseline verification**: Does the search component render without JS errors?
2. **Functional verification**: Does Fuse.js return correct results for known queries?
3. **Edge case verification**: Empty query, no results, special characters
4. **Integration verification**: Does the component work within the Astro page without hydration issues?
